# C60 unlock — SDP over WebHID, with a native fallback

The Polycom Trio C60 (codename `kepler_proto1`, i.MX 8M Mini) unlocks over
i.MX **SDP** (`1fc9:0134`), the BootROM's HID recovery protocol. SDP is
normally driven by **`uuu`**, which ships no WinUSB/WebUSB descriptors, but
the BootROM's vendor HID interface (usage page 0xFF00) is reachable over
**WebHID**, so the whole C60 unlock runs in the browser: WebHID SDP loads
U-Boot into RAM, the device drops into fastboot, and the shared OS-install
steps flash Linux over WebUSB — the same tail as the TC8. No `uuu`, no
driver, no native app.

The native (Tauri) flavor keeps its own SDP path (`uuu` shell-out by
default, pure-Rust `sdp.rs` behind `native_sdp`) because the Tauri webview
has no WebHID.

## The browser path (WebHID)

- **`core/src/protocol/sdp.ts`** — the SDP protocol module (16-byte
  big-endian `SDPCmd`, `WRITE_FILE`/`JUMP`, HID reports 1–4, IVT/BootData
  parsing), written against the seam's `HidTransport` interface.
- **`core/src/transport/transport.ts`** — optional `hidTransport` capability;
  the web backend wraps `navigator.hid`.
- **`core/src/profiles/c60.ts`** — the "Unlock and Install" flow: SDP-load
  the SPL, let its SDPV gadget take the U-Boot FIT, persist that open U-Boot
  into the eMMC boot area, then reuse the shared fastboot install steps.
  Install/Update and Configure use the same fastboot path as TC8 once the
  C60 is running the open U-Boot gadget.

## The native path (Tauri backend)

Layered the same way — the flow is generic, only the backend differs:

- **`core/transport.ts`** — optional `Backend.c60Provision(opts)` capability
  (`flashBin`, `hammerSecs`, `cmds`, `uartPath?`, `onLog`). The web backend
  leaves it undefined; native implements it.
- **`web/src/native/backend.ts`** — `c60Provision` invokes the `c60_provision`
  Tauri command and forwards `c60-progress` events to `onLog`.
- **`native/src-tauri/src/lib.rs`** — `c60_provision` command: stage
  `flash.bin` → wait for SDP `1fc9:0134` (nusb) → load U-Boot → open the UART
  (`serialport`) → hammer CR for `hammerSecs` → send the slot `cmds`. Streams
  `c60-progress` events.

This mirrors the handoff in `polycom-uboot` (the `c60-dualboot`
scripts and `BOOT_RECIPES.md` for the `c60-kepler_proto1` target).

### Two U-Boot-load paths (`native_sdp` flag)

`uuu -b spl flash.bin` on i.MX8MM runs a two-stage load: BootROM **SDP**
(`1fc9:0134`) loads the SPL and jumps; the SPL brings up DDR and re-enumerates
as **SDPV**, which loads the U-Boot FIT into DRAM and jumps.

- **Default (`native_sdp` off): shell out to `uuu`.** The fallback; needs
  `uuu` on PATH or bundled (per-OS bundling required).
- **`native_sdp` on: the pure-Rust SDP loader (`sdp.rs`) — no external
  binary.** One Rust source compiles for all three OSes over `nusb`: no
  bundling, no macOS notarization of a shipped executable. It reimplements
  the SDP wire protocol (constants and structs transcribed from mfgtools
  `libuuu/sdp.{h,cpp}`); the protocol codec and IVT parsing are unit-tested.

The UART driving (recipe-specific, simple) is native Rust over `serialport`
in both paths.

### `native_sdp` — what the pure-Rust path depends on

The `nusb` implementation must match the BootROM's HID behavior: the framing
(interrupt-OUT, with `SET_REPORT` control transfers as the fallback), the
report sizes, and the HAB (report 3) / status (report 4) handshake ordering
around `WRITE_FILE`. It also keys re-enumeration on the SPL's SDPV gadget
PID (`SDP_PID_SPL_SDPV`, `0x0151`; `0x0152` is the final U-Boot fastboot
gadget).

## Prerequisites to run it

1. **For the native flavor only** — either `uuu` on PATH (a shipped app
   needs per-OS `uuu` binaries through `tauri.conf.json`
   `bundle.externalBin`), or the pure-Rust `native_sdp` path, which bundles
   nothing. The browser path has no prerequisite.
2. **`c60-manifest.json`** — a release asset on the **C60 firmware repo**
   (separate from `poly-firmware-build`; the proxy's allowlist maps the
   `c60` device key to it). The wizard fetches it through the proxy at
   `/artifact/c60/<tag>/c60-manifest.json`, and its presence is what makes
   a release appear in the C60 OS chooser. For local dev, drop it (plus
   `flash.bin`) into `packages/web/public/artifacts/`. Shape:
   ```json
   {
     "device": "c60", "version": "v0.1.8",
     "flashbin": { "url": "flash.bin", "size": <bytes>, "sha256": "<sha256>" },
     "hammerSecs": 4,
     "bootSeq": [
       "mmc dev 0",
       "mmc read 0x44000000 0x8000 0x18000",
       "cp.b 0x44000800 0x40080000 0x019f1200",
       "cp.b 0x459f2000 0x46000000 0x0000bf13",
       "setenv bootargs '… root=PARTLABEL=system_a'",
       "booti 0x40080000 - 0x46000000"
     ],
     "os": {
       "boot":   { "url": "boot.img",    "target": "boot_a" },
       "dtbo":   { "url": "dtbo.img",    "target": "dtbo_a" },
       "vbmeta": { "url": "vbmeta.img",  "target": "vbmeta_a" },
       "rootfs": { "url": "rootfs.simg", "target": "system_a", "format": "android-sparse" },
       "gptRestore": null
     }
   }
   ```
   The `mmc read`/`cp.b` addresses are build-specific (they depend on the
   packed kernel size) — regenerate them with each image build. That's
   exactly why they live in the manifest, not the code. `flash.bin` is the
   `c60-kepler_proto1` build output. The **`os` section is optional**: when
   the release carries C60 OS images, list them there and the install tail
   flashes them; when absent, the install steps fail with a clear "no C60
   OS images in this release" error, and unlock still works. All URLs are
   flat release-asset names. The install tail flashes each image to its
   `target` (rootfs defaults to `userdata` for TC8-era manifests without
   one). `"gptRestore": null` declares that no GPT-restore image exists —
   the stock Polycom table is treated as intact and the repair step is skipped
   (a wiped table needs a manual re-partition); a
   `{ "url": "c60-gpt-restore.simg", "diskSectors": … }` entry turns repair
   on.

## Limitations

- **Persistent replacement depends on the boot-area image winning cold
  boot.** The browser flow flashes `flash.bin` to the C60 eMMC boot area
  through the built-in FSL fastboot `bootloader0` target (`bootloader` on
  non-A/B builds) before installing Linux; the flow itself does not confirm
  that a subsequent cold boot comes up in the open U-Boot rather than stock.
