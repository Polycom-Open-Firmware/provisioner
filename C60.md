# C60 unlock — SDP over WebHID, with a native fallback

Status: the **browser path (WebHID SDP) is live** and boots our U-Boot on
hardware. The **native path** (`uuu` shell-out, pure-Rust `sdp.rs` behind
`native_sdp`) compiles and is unit-tested but not hardware-verified.
Started 2026-07-02; updated 2026-07-03.

The Polycom Trio C60 (codename `kepler_proto1`, i.MX 8M Mini) unlocks over
i.MX **SDP** (`1fc9:0134`), the BootROM's HID recovery protocol. This began as
a native-only flow — SDP is normally driven by **`uuu`**, which ships no
WinUSB/WebUSB descriptors — until we proved the BootROM's vendor HID interface
(usage page 0xFF00) is reachable over **WebHID**. The whole C60 unlock now
runs in the browser: WebHID SDP loads U-Boot into RAM, the device drops into
fastboot, and the shared OS-install steps flash Linux over WebUSB — the same
tail as the TC8. No `uuu`, no driver, no native app.

The native (Tauri) flavor keeps its own SDP path because the Tauri webview
has no WebHID.

## The browser path (WebHID)

- **`core/src/protocol/sdp.ts`** — the SDP protocol module (16-byte
  big-endian `SDPCmd`, `WRITE_FILE`/`JUMP`, HID reports 1–4, IVT/BootData
  parsing), written against the seam's `HidTransport` interface.
- **`core/src/transport/transport.ts`** — optional `hidTransport` capability;
  the web backend wraps `navigator.hid`.
- **`core/src/profiles/c60.ts`** — the "Unlock and Install" flow: SDP-load
  the SPL, let its SDPV gadget take the U-Boot FIT, then reuse the shared
  fastboot install steps. Install/Update and Configure are the identical
  TC8 flows, reused verbatim (post-boot, the C60 runs the same open U-Boot
  fastboot gadget).

## The native path (Tauri backend)

Layered the same way — the flow is generic, only the backend differs:

- **`core/transport.ts`** — optional `Backend.c60Provision(opts)` capability
  (`flashBin`, `hammerSecs`, `cmds`, `uartPath?`, `onLog`). The web backend
  leaves it undefined; native implements it.
- **`web/src/native/backend.ts`** — `c60Provision` invokes the `c60_provision`
  Tauri command and forwards `c60-progress` events to `onLog`.
- **`native/src-tauri/src/lib.rs`** — `c60_provision` command: stage
  `flash.bin` → wait for SDP `1fc9:0134` (nusb) → load U-Boot → open the UART
  (`serialport`) → hammer CR for `hammerSecs` → send the slot `cmds`. Streams
  `c60-progress` events.

This mirrors the proven handoff in `polycom-uboot` (the `c60-dualboot`
scripts and `BOOT_RECIPES.md` for the `c60-kepler_proto1` target).

### Two U-Boot-load paths (`native_sdp` flag)

`uuu -b spl flash.bin` on i.MX8MM runs a two-stage load: BootROM **SDP**
(`1fc9:0134`) loads the SPL and jumps; the SPL brings up DDR and re-enumerates
as **SDPV**, which loads the U-Boot FIT into DRAM and jumps.

- **Default (`native_sdp` off): shell out to `uuu`.** Proven; needs `uuu` on
  PATH or bundled per OS (the bundling pain). Kept as the fallback.
- **`native_sdp` on: the pure-Rust SDP loader (`sdp.rs`) — no external
  binary.** One Rust source compiles for all three OSes over `nusb`: no
  bundling, no macOS notarization of a shipped executable. It reimplements
  the SDP wire protocol (constants and structs transcribed from mfgtools
  `libuuu/sdp.{h,cpp}`); the protocol codec and IVT parsing are unit-tested
  (`cargo test --lib sdp::`, 4 tests green).

The UART driving (recipe-specific, simple) is native Rust over `serialport`
in both paths.

### `native_sdp` — what still needs hardware to confirm

The browser (WebHID) path already validated the protocol ordering on a real
C60 — SPL load, SDPV re-enumeration, FIT load, jump. What remains is specific
to the native `nusb` implementation. Before flipping `native_sdp` on by
default:

- **HID framing over `nusb`** — interrupt-OUT vs `SET_REPORT` control
  fallback, report sizes, and the HAB (report 3) / status (report 4)
  handshake ordering around `WRITE_FILE`.
- **SDPV re-enumeration PID** (`SDP_PID_SPL_SDPV`, currently `0x0152`) —
  confirm `nusb` sees the same PID the browser does.

Until then, `uuu` (the default) is how the native flavor runs on a bench;
the browser path needs no `uuu` at all.

## Prerequisites to run it

1. **For the native flavor only** — either `uuu` on PATH (present on the dev
   host; a shipped app would need per-OS `uuu` binaries through
   `tauri.conf.json` `bundle.externalBin`), or validate the pure-Rust
   `native_sdp` path so nothing is bundled at all. The browser path has no
   prerequisite.
2. **`c60-manifest.json` artifact** served alongside the others, shape:
   ```json
   {
     "flashbin": { "url": "c60/flash.bin" },
     "hammerSecs": 14,
     "bootSeq": {
       "a": [
         "mmc dev 0",
         "mmc read 0x44000000 0x8000 0x18000",
         "cp.b 0x44000800 0x40080000 0x019f4a00",
         "cp.b 0x459f5800 0x46000000 0xb79d",
         "setenv bootargs 'console=tty0 console=ttymxc1,115200 … root=/dev/disk/by-partlabel/system_a'",
         "booti 0x40080000 - 0x46000000"
       ]
     }
   }
   ```
   The `mmc read`/`cp.b` addresses are build-specific (they depend on the
   packed kernel size) — regenerate them with each image build. That's
   exactly why they live in the manifest, not the code. `flash.bin` is the
   `c60-kepler_proto1` build output.

## Known limitations (next phase)

- **DRAM boot, not persistent.** Each unlock loads U-Boot from the host;
  autonomous autoboot needs a `bootcmd` macro persisted to the environment
  (a `boot_slot` selector) — a firmware-side TODO.
- The native path has not run on hardware: verify SDP detection, `uuu`
  invocation, and the UART recipe against a real C60 before shipping the
  Tauri flavor.
